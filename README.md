# Math Battle Online

数字を弾として撃ち合う、デッキ構築型2Dターン制対戦ゲーム。

## ゲーム概要

プレイヤーはデッキからカード (数字・演算子・移動) を引き、「計算」で数字を合成し、「攻撃」で相手に数字の弾を飛ばします。
さらに「関数」で数学的な曲線をフィールドに描いてダメージゾーンを作ったり、フィールドにスポーンするアイテムを撃破/接触で獲得することも可能。
画面構成は2D格闘ゲーム風で、相手のHPを0にすれば勝利です。

### 基本ルール

| 項目 | 内容 |
|------|------|
| 対戦形式 | オンライン1vs1 |
| ターン制 | 同時入力 → 同時解決 |
| デッキ | 演算子 (+, -, ×, ÷) で構築。数字・移動カードは手札補充で自動供給 |
| デッキ枚数 | 5〜20枚、同名カードは最大6枚まで |
| 手札 | ターン開始時にデッキから2枚ドロー (上限16枚) |
| アクション | 即時 (移動カード使用 / 計算) と メイン (攻撃 / 関数 / スキップ) の2系統 |
| HP | 50 |

### アクション

**即時アクション (1ターンに何度でも実行可)**
- **移動カード使用** — 手札の移動カードを消費して上下左右に40px移動
- **計算** — 手札の数字・演算子を3〜5枚消費して新しい数値トークンを合成 (例: `3 + 7` → `10`)

**メインアクション (1ターンに1つだけ・送信でターン解決)**
- **攻撃** — 手札の数字を正面に弾として発射
- **関数** — `f(x)` を定義してフィールドに曲線を描く。曲線上の敵に毎ターン10ダメージ (1人10回まで)
- **スキップ** — メインアクションを行わない

### 弾の特徴

- 大きい数字ほど高威力だが速度が遅い (`speed = 80 / (1 + value × 0.15)`)
- 弾同士の衝突: 大きい方の値 - 小さい方の値 = 残る弾の値 (同値なら両方消滅)
- **素数弾 (10以上の素数) は特殊**: 通常弾を貫通しながら値を削り、自身は減らない
- 壁に反射するたびに数字が +3 (最大3回まで反射、超過で消滅)

### アイテム

毎ターン一定確率でフィールドに出現する獲得可能オブジェクト。両プレイヤーが弾で攻撃可能で、最後にHPを0にした側が獲得。プレイヤーが触れた場合も即時拾得。

| 種別 | 効果 |
|------|------|
| `+` `-` `×` `÷` | 対応する演算子カード1枚を手札に追加 |
| `pack` | 演算子4種をまとめて手札に追加 |
| `heal` | HP を 5〜20 回復 (上限は初期HP) |

## クイックスタート

### 必要なもの
- Node.js 20以上

### セットアップ

```bash
# クローン & インストール
git clone <repo-url>
cd math-battle
npm install

# 開発サーバー起動 (ターミナル1つで両方起動)
npm run dev:all

# または個別に起動 (ターミナル2つ)
npm run dev      # Next.js → localhost:3000
npm run server   # WebSocket サーバー → localhost:1999
```

ブラウザのタブを2つ開いて `http://localhost:3000` にアクセスすればローカルで対戦テストできます。

### デプロイ

スマホで遊ぶための公開デプロイは **Vercel + Render** の構成です。
詳細手順は [`DEPLOYMENT.md`](./DEPLOYMENT.md) を参照してください。

- フロントエンド (Next.js) → **Vercel**
- WebSocketサーバー → **Render** (`render.yaml` の Blueprint で自動デプロイ可能)
- 環境変数 `NEXT_PUBLIC_WS_URL` で接続先を指定 (例: `wss://mathbattle-ws.onrender.com`)

## アーキテクチャ

```
ブラウザ A ──WebSocket──┐
                        ├── ws サーバー (Render) ── (将来: Supabase)
ブラウザ B ──WebSocket──┘
```

- **フロントエンド**: Next.js 15 (App Router), TypeScript, Tailwind CSS v4
- **リアルタイム同期**: 自前の WebSocket サーバー (`ws` パッケージ, `server/index.ts`)
- **データベース & 認証**: 未実装 (将来 Supabase + Google OAuth を予定)

全ゲームロジックはサーバー上で動作します (サーバー権威型)。クライアントは表示と入力のみ。

> **メモ**: 当初は PartyKit を採用予定でしたが、Windows 環境のパスバグにより自前の `ws` サーバーへ切り替えました。`party/index.ts` は参照用に残しています。

## プロジェクト構成

```
app/              → ページ (ロビー + デッキ構築 / ゲームルーム)
components/       → React コンポーネント (game/ と lobby/)
lib/              → 共有ロジック (型定義・カード・計算・関数・物理・アイテム・座標等)
server/           → WebSocket サーバー (コアゲームループ)
hooks/            → カスタム React フック (useGameSocket)
```

## 設定ファイル

### `game-config.json` — 動的バランス調整 (サーバー再起動で反映)

プロジェクトルートに置くと WebSocket サーバー起動時に読み込まれます。ファイルがない場合はデフォルト値が使われます。

```json
{
  "actionTimeoutSec": 45
}
```

| キー | デフォルト | 説明 |
|------|-----------|------|
| `actionTimeoutSec` | `45` | アクション選択フェーズのタイムアウト (秒)。`0` 以下で時間制限なし |

### `lib/constants.ts` — ゲームバランス定数 (コード変更・再ビルドで反映)

サイズ・速度・HP などゲームの根幹となる数値はここで一元管理されています。

```ts
// プレイヤー
PLAYER_SIZE = 24      // 当たり判定の半径 (px)。見た目と判定が同時に変わる
INITIAL_HP = 50       // 初期HP
MOVE_DISTANCE = 40    // 移動カード1枚あたりの移動距離 (px)

// 弾
BULLET_SIZE = 10            // 当たり判定の半径 (px)
BASE_BULLET_SPEED = 80      // 弾の基準速度
SPEED_DECAY_FACTOR = 0.15   // 大きい数字ほど遅くなる係数
WALL_REFLECTION_BONUS = 3   // 壁反射ごとに増加する数値
MAX_REFLECTIONS = 3         // 最大反射回数

// フィールド
FIELD_WIDTH = 800     // 論理サイズ (px)
FIELD_HEIGHT = 400

// 手札・デッキ
DRAW_COUNT = 2                    // 1ターンにドローする枚数
MAX_HAND_SIZE = 16                // 手札上限
MAX_CALC_CARDS = 5                // 計算に使えるカードの最大枚数
NUMBER_REPLENISH_THRESHOLD = 3    // 数字カードがこの枚数以下なら 1〜9 を自動補充
MOVE_AUTO_REPLENISH = true        // 移動カードを各方向毎ターン自動補充
MIN_DECK_SIZE = 5                 // デッキ最小枚数
MAX_DECK_SIZE = 20                // デッキ最大枚数
MAX_SAME_CARD_COUNT = 6           // 同名カードの投入上限

// 関数アクション
MAX_FUNCTION_USES = 10    // 関数アクションの使用回数上限 (プレイヤーごと)
FUNCTION_DAMAGE = 10      // 曲線ダメージ (HP/ターン)

// アイテム
ITEM_SIZE = 40            // 当たり判定の直径 (px)
MAX_ITEMS = 5             // フィールド同時存在上限
ITEM_HP_MIN = 1           // ランダムHP下限
ITEM_HP_MAX = 50          // ランダムHP上限

// 背景グリッド (数学座標単位)
GRID_SPACING_X = 0.5
GRID_SPACING_Y = 0.5
```

> **NOTE:** `PLAYER_SIZE` と `BULLET_SIZE` は当たり判定の半径であるとともに、スプライトの描画サイズにも直結しています。変更すると物理的な判定と見た目が同時に変わります。
